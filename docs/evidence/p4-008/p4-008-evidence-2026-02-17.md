### Automated Scoring Pipeline Design

```
Scoring Pipeline (to be implemented at activation):

1. Input: Provider transcript + Ground truth transcript (per encounter)

2. Preprocessing:
   - Normalize text: lowercase, remove punctuation, expand abbreviations
   - Tokenize: word-level tokenization
   - Align: dynamic time warping alignment of transcript to reference

3. Standard WER Computation:
   - Use jiwer library (Python) for WER calculation
   - Compute: substitutions, insertions, deletions, total reference words
   - Output: WER per encounter, mean WER, std dev, 95% CI

4. Medical WER Computation:
   - Filter tokens: retain only tokens matching OMOP concept synonyms
     OR RxNorm ingredient/brand names
     OR SNOMED body structure terms
   - Compute WER on filtered token set only
   - Output: Medical WER per encounter, mean, std dev, 95% CI

5. Per-Specialty Breakdown:
   - Group encounters by specialty
   - Compute WER and Medical WER per specialty
   - Flag any specialty exceeding per-specialty threshold

6. Error Categorization:
   - Medical terminology substitution (e.g., metformin -> metaforming)
   - Drug name error (e.g., lisinopril -> lysin a pearl)
   - Dosage/number error (e.g., 40mg -> 14 mg)
   - Abbreviation error (e.g., BID -> bid)
   - Speaker attribution error (clinician/patient confused)
   - Background noise artifact (non-speech -> words)
   - Other

7. Report Generation:
   - Per-provider comparison table (WER, Medical WER, latency, cost)
   - Per-specialty heatmap
   - Error category distribution
   - Statistical significance tests (paired t-test between providers)
   - Recommendation with confidence level
```

## PHI Compliance Validation Checklist

### Consent Capture

- [ ] Recording consent captured before any audio recording begins
- [ ] Consent UI displays clear disclosure of recording purpose, storage, and processing
- [ ] Consent is explicit opt-in (not opt-out or implied)
- [ ] Separate consent captured for cloud STT processing (if applicable)
- [ ] Consent record includes: patient ID, encounter ID, timestamp, consent version, clinician witness
- [ ] Consent withdrawal mechanism exists and triggers audio deletion
- [ ] Consent audit trail is tamper-evident

### Encryption at Rest

- [ ] Audio files encrypted with AES-256
- [ ] Encryption uses FIPS 140-2 validated cryptographic module
- [ ] Encryption keys stored separately from encrypted data
- [ ] Key rotation schedule defined (minimum annual)
- [ ] Encrypted audio cannot be read without decryption key (verified by direct storage inspection)
- [ ] Transcripts encrypted at rest with same standard as audio

### BAA Verification

- [ ] BAA executed with STT provider before any PHI audio is processed
- [ ] BAA scope covers: audio transmission, processing, temporary storage, breach notification
- [ ] BAA specifies provider must not retain audio beyond processing window
- [ ] BAA includes breach notification within 24 hours
- [ ] BAA grants audit rights over provider PHI handling
- [ ] BAA discloses all subcontractors handling PHI
- [ ] BAA termination clause requires data destruction within 30 days
- [ ] N/A if self-hosted STT selected (document rationale)

### Access Audit

- [ ] Every audio file access generates audit event (play, download, process, delete)
- [ ] Every transcript access generates audit event (read, edit, approve, export)
- [ ] Audit events include: timestamp, user identity, action, resource ID, IP address
- [ ] Audit logs stored separately from operational data
- [ ] Audit logs retained for minimum 7 years
- [ ] Audit log integrity verified (tamper-evident, append-only)
- [ ] Unauthorized access attempts logged and alerted

### Retention Policy

- [ ] Raw audio retention period set to 7 years (aligned with P1-028 clinical note retention)
- [ ] Transcripts retention period set to 7 years
- [ ] Processing logs retention period set to 3 years
- [ ] Automated retention enforcement: audio older than retention period flagged for secure disposal
- [ ] Secure disposal method: cryptographic erasure or verified overwrite per NIST SP 800-88
- [ ] Retention policy documented and reviewed by Compliance

## End-to-End Accuracy Measurement Plan

### Combined Clinical Accuracy (CCA) Metric

```
CCA = (1 - Medical_WER) * Section_Detection_Accuracy * Entity_Extraction_F1

Where:
  Medical_WER = Word Error Rate on OMOP-filtered medical terminology tokens
  Section_Detection_Accuracy = % of transcript segments assigned to correct clinical section
  Entity_Extraction_F1 = F1 score of mention extraction on voice-sourced text vs. gold standard

Target: CCA > 0.85 (85%)
```

### Measurement Protocol

| Step | Activity | Metric | Target |
|------|----------|--------|--------|
| 1 | STT transcribes 100 encounters | Medical WER | <5% overall, <4% per specialty |
| 2 | Section detection labels transcript segments | Section accuracy | >90% segments correctly labeled |
| 3 | NLP extraction on voice text | Entity F1 | >85% F1 (condition, medication, procedure mentions) |
| 4 | OMOP mapping of extracted mentions | Mapping accuracy | >90% of mentions mapped to correct OMOP concept |
| 5 | Clinical note comparison | Note similarity | >80% semantic similarity to gold-standard typed note |
| 6 | Clinician review | Correction rate | <10% of note sections require correction |
| 7 | Combined CCA computation | CCA | >85% |
